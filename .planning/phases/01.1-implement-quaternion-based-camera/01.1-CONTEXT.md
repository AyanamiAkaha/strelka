# Phase 1.1: Implement Quaternion-Based Camera - Context

**Gathered:** 2026-02-02
**Status:** Ready for planning

<domain>
## Phase Boundary

Implement quaternion-based camera rotation to eliminate gimbal lock. Replace the existing Euler angle-based camera implementation with quaternion logic while preserving shader-based calculations. This phase delivers a camera system that correctly handles freeflight rotation in all axes without coordinate system collapse at extreme angles.

</domain>

<decisions>
## Implementation Decisions

### API transition strategy
- Clean replacement of Camera.ts internals (not side-by-side migration)
- Preserve shader calculations — choose whichever API approach (lookAt vs quaternion) keeps shader code intact
- Remove Euler angle calculations entirely once quaternion implementation is working
- Claude's discretion on whether changes needed in other files referencing Camera.ts

### Rotation behavior & feel
- Keep same rotation sensitivity (degrees per pixel) as current implementation
- Instant response to input (no damping/momentum)
- Pitch and yaw only (no roll control, same as current)
- Single rotation speed (no configurable presets)

### State preservation
- Reset to default camera position and orientation (not preserve current state)
- Use easiest default "zero" position and rotation — this is for initialization and 'R' key reset
- Use camera's actual local up vector (not fixed global up (0, 1, 0))
- No backward compatibility needed — choose easiest approach that maintains shader calculations and freeflight mode

### Testing & validation approach
- Manual testing by user (not automated unit tests)
- Test both scenarios: extreme angles (±90°) to confirm no gimbal lock, and complex multi-axis rotation sequences
- Test quaternion camera independently (no comparison with Euler camera output)
- Focus testing on Camera.ts quaternion logic — rely on manual testing for shader integration; quaternions are simple matrix operations, mistakes are easy to spot

### Claude's Discretion
- Whether other parts of codebase need updates for Camera API changes (breaking changes acceptable)
- Choice between preserving lookAt API vs switching to position + rotation quaternion (prioritize shader calculation preservation)

</decisions>

<specifics>
## Specific Ideas

- **Freeflight mode is critical**: rotation and movement always in camera space, no objective "up" and "down" — global vectors are technical details, local ones are what matter
- **No backward compatibility needed**: reset to default on initialization and 'R' key, no need to migrate state
- **Shader preservation prioritized**: keep as much calculation in shader as possible
- **Quaternions are straightforward**: simple matrix operations, relatively easy to identify mistakes during manual testing

</specifics>
