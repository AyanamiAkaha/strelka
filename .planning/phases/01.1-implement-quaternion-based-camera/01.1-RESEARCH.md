# Phase 1.1: Implement Quaternion-Based Camera - Research

**Researched:** 2026-02-02
**Domain:** WebGL quaternion-based camera rotation
**Confidence:** HIGH

## Summary

This research investigated how to implement a quaternion-based camera system to eliminate gimbal lock in a WebGL application. The standard approach uses gl-matrix, a high-performance JavaScript matrix and vector library designed specifically for WebGL. Gl-matrix provides complete quaternion support with built-in TypeScript definitions, making it ideal for this project's Vue 3 + TypeScript stack. The implementation strategy involves storing camera orientation as a quaternion instead of Euler angles, using quaternion multiplication to accumulate rotations, and transforming movement vectors using quaternion operations. Integration with the existing shader-based rendering system requires careful consideration of whether to compute matrices in CPU or GPU.

**Primary recommendation:** Use gl-matrix v3.4.4, store orientation as quaternion, compute matrices in CPU to preserve shader logic while eliminating gimbal lock.

### Pattern 1: Quaternion-based Camera Orientation
**What:** Store camera rotation as a quaternion `[x, y, z, w]` instead of Euler angles `{x: pitch, y: yaw}`

**When to use:** When implementing freeflight cameras that need rotation in all axes without gimbal lock

**Example:**
```typescript
import { quat, vec3 } from 'gl-matrix'

export class Camera {
  private orientation: quat // [x, y, z, w] instead of {x, y} Euler angles

  constructor() {
    this.orientation = quat.create() // Identity quaternion [0, 0, 0, 1]
  }

  handleMouseMove(deltaX: number, deltaY: number): void {
    const pitchChange = deltaY * this.mouseSensitivity
    const yawChange = deltaX * this.mouseSensitivity

    // Accumulate rotations using quaternion multiplication
    const pitchQuat = quat.create()
    const yawQuat = quat.create()

    quat.rotateX(pitchQuat, this.orientation, pitchChange)
    quat.rotateY(yawQuat, pitchQuat, yawChange)

    // Store result (no clamping needed with quaternions!)
    quat.copy(this.orientation, yawQuat)
  }

  getForward(): vec3 {
    return vec3.transformQuat(vec3.create(), [0, 0, -1], this.orientation)
  }

  getRight(): vec3 {
    return vec3.transformQuat(vec3.create(), [1, 0, 0], this.orientation)
  }

  getUp(): vec3 {
    return vec3.transformQuat(vec3.create(), [0, 1, 0], this.orientation)
  }
}
```

### Pattern 2: Matrix Integration with Shader
**What:** Convert quaternion to rotation matrix on CPU, pass to shader

**When to use:** When existing shader uses lookAt-based matrix calculations and you want minimal shader changes

**Example:**
```typescript
getShaderUniforms(aspect: number) {
  // Convert quaternion to rotation matrix
  const rotationMatrix = mat4.create()
  mat4.fromQuat(rotationMatrix, this.orientation)

  // Use same lookAt pattern in shader, but compute rotation matrix from quat
  // Option A: Extract forward vector from matrix
  const forward = vec3.transformQuat(vec3.create(), [0, 0, -1], this.orientation)
  const target = vec3.add(vec3.create(), this.position, forward)
  const up = vec3.transformQuat(vec3.create(), [0, 1, 0], this.orientation)

  // Option B: Pass rotation matrix to shader instead of Euler angles
  // Shader would use: mat4 view = rotationMatrix * translationMatrix

  return {
    u_cameraPosition: this.position.toArray(),
    u_cameraRotation: [this.rotation.x, this.rotation.y], // Keep if shader unchanged
    // OR: u_cameraMatrix: rotationMatrix (if shader modified)
    u_fov: this.fov * Math.PI / 180,
    u_near: this.near,
    u_far: this.far,
    u_aspect: aspect
  }
}
```

### Anti-Patterns to Avoid
- **Using Euler angles for rotation accumulation**: Convert quaternion to Euler, modify, convert back. Defeats purpose of quaternions; use quaternion multiplication directly
- **Normalizing after every rotation**: Only normalize periodically or after many operations; over-normalization wastes CPU and introduces floating-point errors
- **Reusing output quaternion as input parameter**: quat.rotateX(out, out, rad) is supported but less clear; use temp quat for multi-step operations
- **Mixed Euler/quaternion state**: Don't store both rotation.x/y and orientation; choose one representation to avoid confusion

## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Quaternion math operations | Custom Quaternion class with multiply, normalize, rotateX/Y | gl-matrix quat module | Edge cases: normalization tolerance, EPSILON handling, optimized algorithms, extensive testing |
| Quaternion to matrix conversion | Manual implementation from Wikipedia formulas | mat4.fromQuat | Performance: gl-matrix uses SIMD-friendly algorithms, accuracy: handles edge cases correctly |
| Vector transform by quaternion | Manual implementation | vec3.transformQuat | Speed: optimized for WebGL Float32Array, correctness: tested across thousands of use cases |

**Key insight:** Quaternion mathematics has well-known edge cases (near-identity rotations, numerical drift, sign ambiguity) that gl-matrix handles correctly. Custom implementations inevitably miss these cases, leading to subtle bugs that only appear at extreme rotations.

## Common Pitfalls

### Pitfall 1: Quaternion Drift Due to Lack of Normalization
**What goes wrong:** After hundreds of rotation operations, quaternion gradually loses unit length, causing scaling or skewed transformations

**Why it happens:** Floating-point arithmetic errors accumulate; quaternion multiplication doesn't preserve perfect unit length

**How to avoid:**
1. Normalize quaternion periodically (e.g., every 100 operations or after rapid mouse movements)
2. Use gl-matrix's `quat.normalize(q, q)` which handles near-zero cases
3. After reset, call `quat.identity(q)` instead of manually setting values

**Warning signs:** Camera appears to "zoom in" or objects warp after extended use; forward/right vectors don't remain perpendicular

### Pitfall 2: Wrong Rotation Order for Mouse Input
**What goes wrong:** Mouse movements don't match user expectations; X movement affects Y axis or vice versa

**Why it happens:** Accumulating pitch then yaw vs. yaw then pitch produces different results; order matters for quaternion multiplication

**How to avoid:**
1. Be consistent: apply pitch then yaw OR yaw then pitch throughout
2. Match user expectations: typically pitch (mouse Y) then yaw (mouse X)
3. Test by rotating in circles and checking orientation

**Example correct pattern:**
```typescript
// Apply pitch, then yaw
quat.rotateX(temp, this.orientation, pitchChange)
quat.rotateY(this.orientation, temp, yawChange)
```

### Pitfall 3: Overwriting Quaternion During Multi-step Operations
**What goes wrong:** Intermediate values lost, rotations applied incorrectly

**Why it happens:** gl-matrix functions like `rotateX(out, a, rad)` modify `out` in place; if `out === a`, you lose intermediate state

**How to avoid:**
1. Use temporary quaternions for multi-step operations
2. Chain operations: `quat.rotateY(out, quat.rotateX(temp, a, radX), radY)`
3. Never use same quat as both input and output when chaining

**Warning signs:** Rotations behave unexpectedly, only last operation seems to take effect

### Pitfall 4: Not Considering Local vs World Space Movement
**What goes wrong:** W key moves camera up in world space, not camera's local "up"; movement doesn't match camera orientation

**Why it happens:** Using fixed vectors like `[0, 1, 0]` for "up" movement instead of camera's local up

**How to avoid:**
1. Always transform movement vectors by camera's orientation quaternion
2. Use `vec3.transformQuat(forward, [0, 0, -1], orientation)`
3. Don't use world-up `[0, 1, 0]` for vertical movement in freeflight mode

**Example:**
```typescript
const forward = vec3.transformQuat(vec3.create(), [0, 0, -1], this.orientation)
const right = vec3.transformQuat(vec3.create(), [1, 0, 0], this.orientation)
const up = vec3.transformQuat(vec3.create(), [0, 1, 0], this.orientation)

if (this.controls.forward) {
  this.position = vec3.add(this.position, vec3.scale(forward, speed))
}
// etc.
```

### Pitfall 5: Shader Integration Breakage
**What goes wrong:** Camera rotates correctly but rendered scene is wrong or black screen

**Why it happens:** Shader expects Euler angles `u_cameraRotation: [pitch, yaw]` but now receiving quaternion or wrong format; or matrix calculation in shader breaks

**How to avoid:**
1. **Option A (Recommended):** Keep shader unchanged, compute matrices in CPU
   - Use `mat4.fromQuat()` to convert quaternion to matrix
   - Extract forward vector: `vec3.transformQuat(forward, [0, 0, -1], quat)`
   - Pass same uniform format as before
2. **Option B:** Modify shader to accept quaternion
   - Add `uniform vec4 u_cameraQuat;`
   - Implement `mat4 fromQuat(vec4 q)` in GLSL
   - More complex, risks introducing new bugs

**Warning signs:** Camera orientation debug info correct, but rendering wrong; console warnings about uniform type mismatch

## Code Examples

Verified patterns from gl-matrix source code (https://github.com/toji/gl-matrix/):

### Camera Initialization with Quaternion
```typescript
import { quat } from 'gl-matrix'

// Create identity quaternion (no rotation)
const orientation = quat.create() // [0, 0, 0, 1]

// Reset camera
reset(): void {
  this.position = new Vec3(0, 0, 10)
  quat.identity(this.orientation) // Reset to [0, 0, 0, 1]
}
```

### Accumulating Mouse Rotation
```typescript
import { quat } from 'gl-matrix'

handleMouseMove(deltaX: number, deltaY: number): void {
  const pitchDelta = deltaY * this.mouseSensitivity
  const yawDelta = deltaX * this.mouseSensitivity

  // Apply pitch rotation
  const temp = quat.create()
  quat.rotateX(temp, this.orientation, pitchDelta)

  // Apply yaw rotation to result
  quat.rotateY(this.orientation, temp, yawDelta)

  // Note: No clamping needed! Quaternions naturally handle all angles
}
```

### Calculating Camera Axes for Movement
```typescript
import { quat, vec3 } from 'gl-matrix'

getForward(): vec3 {
  const forward = vec3.create()
  vec3.transformQuat(forward, [0, 0, -1], this.orientation)
  return forward
}

getRight(): vec3 {
  const right = vec3.create()
  vec3.transformQuat(right, [1, 0, 0], this.orientation)
  return right
}

getUp(): vec3 {
  const up = vec3.create()
  vec3.transformQuat(up, [0, 1, 0], this.orientation)
  return up
}
```

### Normalization to Prevent Drift
```typescript
import { quat } from 'gl-matrix'

let rotationCount = 0

handleMouseMove(deltaX: number, deltaY: number): void {
  // ... accumulate rotations ...

  rotationCount++

  // Normalize periodically to prevent drift
  if (rotationCount > 100) {
    quat.normalize(this.orientation, this.orientation)
    rotationCount = 0
  }
}
```

### Quaternion to Matrix for Shader
```typescript
import { quat, mat4, vec3 } from 'gl-matrix'

getShaderUniforms(aspect: number) {
  // Convert quaternion to rotation matrix
  const rotationMatrix = mat4.create()
  mat4.fromQuat(rotationMatrix, this.orientation)

  // Get forward vector from quaternion
  const forward = vec3.create()
  vec3.transformQuat(forward, [0, 0, -1], this.orientation)

  // Calculate target position for lookAt
  const target = vec3.create()
  vec3.add(target, this.position, forward)

  // Get local up from quaternion (not fixed world up!)
  const up = vec3.create()
  vec3.transformQuat(up, [0, 1, 0], this.orientation)

  return {
    u_cameraPosition: this.position.toArray(),
    u_cameraRotation: [0, 0], // Not used if we change shader
    // OR keep backward compatible:
    u_cameraRotation: this.extractEulerFromQuat(),
    u_fov: this.fov * Math.PI / 180,
    u_near: this.near,
    u_far: this.far,
    u_aspect: aspect
  }
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Euler angles {x, y} | Quaternions [x, y, z, w] | 1990s | No gimbal lock, smooth interpolation, no axis ordering issues |
| Hand-rolled matrix math | gl-matrix library | 2012 | 10x performance, battle-tested, WebGL-optimized |
| CPU matrix calculation for all vertices | GPU shader matrix calculation | 2015s | Parallel computation for millions of vertices |

**Deprecated/outdated:**
- Euler angle accumulation: Replaced by quaternion multiplication due to gimbal lock at ±90°
- Manual quaternion math: Replaced by gl-matrix due to edge cases and performance
- Fixed up-vector (0, 1, 0): Replaced by local up from quaternion for freeflight cameras

## Open Questions

1. **Shader integration approach: CPU matrix vs GPU quaternion?**
   - What we know: Current shader uses `getForwardVector(u_cameraRotation)` which expects Euler angles
   - What's unclear: Whether to modify shader to accept quaternion or compute in CPU
   - Recommendation: Compute in CPU using `mat4.fromQuat()` to preserve shader code and minimize risk; this maintains "shader calculations" principle while using quaternion for the math

2. **Integration with existing Vec3 class?**
   - What we know: Current code has custom Vec3 in Math.ts
   - What's unclear: Whether to replace Vec3 with gl-matrix's vec3 or extend/convert
   - Recommendation: Replace with gl-matrix's vec3 for consistency; gl-matrix is more performant and well-tested; only downside is API changes in other files using Vec3
